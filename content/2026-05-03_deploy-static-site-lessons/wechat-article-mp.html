从蓝盾 CI/CD 到本地脚本

部署个人静态网站的踩坑实录

上周折腾了一件事：给自己的个人网站搭一套自动部署。听起来很简单对吧？静态站点，rsync 同步到服务器，Nginx 一刷新就完事了。

结果我在 SSH 连接这一个环节上卡了整整一个晚上，排查了 6 轮，走过了从"应该很快搞定"到"要不算了"的完整心路历程。

今天把这个过程完整记录下来，希望能帮到遇到类似问题的你。

 

📋 背景

项目：纯静态个人网站（HTML/CSS/JS，零构建依赖）

代码托管：工蜂（公司内部 Git）

服务器：腾讯云轻量应用服务器，Ubuntu，公网 IP

域名 + CDN：EdgeOne 代理 HTTPS，回源到 Nginx 80 端口

目标：push 代码后自动部署到服务器

 

🔧 方案一：蓝盾 CI/CD 流水线

第一反应是用公司的蓝盾（BK-CI）。思路很清晰：

阶段 内容
触发 监听 master 分支的 push 事件
构建&部署 SSH + rsync 同步文件到服务器
验证 curl 健康检查

看起来很简单吧？然后噩梦就开始了。

 

💥 六轮踩坑实录

坑 1：蓝盾变量系统不支持凭证类型

蓝盾有凭证管理功能，我创建了一个 SSH 私钥凭证。但在经典模式流水线中，变量类型下拉框里只有字符串、文本框、布尔值——没有"凭证"选项。

解决：用「文本框」变量替代，把 SSH 私钥内容直接粘贴为变量值，勾选「敏感」选项。

💡 经验：不同蓝盾版本的功能差异很大。如果你用的是智研/经典模式，很多高级功能（如凭证引用）可能不可用。提前确认再动手。

坑 2：多行私钥被文本框变量吃掉换行符

第一次运行流水线就报错了：kex_exchange_identification: Connection closed by remote host

排查发现，蓝盾的文本框变量在注入到 Bash 脚本时，多行内容的换行符被破坏了。SSH 私钥是严格的 PEM 格式，每 64 字符必须换行，格式一破坏就无法使用。

解决：把私钥做 Base64 编码后存入变量（变成一行），脚本中用 base64 -d 解码还原。

# Mac 上编码
base64 < key.pem | tr -d '\n' | pbcopy

# 脚本中解码
echo "${deploy_ssh_key}" | base64 -d > ~/.ssh/deploy_key

💡 经验：CI/CD 系统中传递多行敏感内容，Base64 编码是最稳的方案。不要相信任何平台的"文本框"能完整保留换行符。

坑 3：服务器遭受 SSH 暴力破解攻击

修复了密钥格式后，还是同一个错误。查看服务器的 auth.log，发现了一个可怕的事实：

Failed password for invalid user admin from 204.76.203.233
Failed password for root from 200.89.69.247
Failed password for invalid user admin from 102.88.137.213
Connection reset by 113.250.184.183 [preauth]
Connection closed by 200.196.50.91 [preauth]
...

服务器 22 端口裸露在公网上，平均每分钟都有陌生 IP 在尝试 SSH 登录。当时初步怀疑是攻击流量占满了 SSH 的 MaxStartups 并发连接限制，导致蓝盾构建机被"挤掉"了。

处理：紧急做了三件事——

措施 作用
安装 fail2ban 3 次密码错误自动封禁 IP 1 小时
MaxStartups 调至 30:50:80 并发连接容量提升 3 倍
PasswordAuthentication no 禁止密码登录，暴力破解彻底失效

⚠️ 警告：任何公网服务器的 22 端口都会被扫描。如果你只用密钥登录，第一时间关掉密码认证。这不是可选项，而是必选项。

坑 4：加固后问题依旧——排除了所有服务端原因

做完安全加固后，满怀信心地重新触发流水线……结果还是 kex_exchange_identification: Connection closed by remote host。

这次进行了更深入的排查：

排查项 结果
TCP 端口可达？ ✅ 可达
当前 SSH 连接数 ✅ 只有 1 个（不是并发问题）
密钥格式 ✅ PEM RSA private key，权限 600
hosts.deny / AllowUsers ✅ 无任何限制
Kex 算法兼容 ✅ 服务器支持多种算法
authorized_keys ✅ 公钥已在

所有服务端因素全部排除。问题一定出在客户端（蓝盾构建机）到服务器的网络链路上。

坑 5：换到开发容器也一样——怀疑网络层面问题

放弃蓝盾后，我在 AnyDev 开发容器里写了个部署脚本直接执行——居然也报同一个错！

关键转折点来了。用 ssh -v 开启详细日志模式，真相终于浮出水面：

debug1: Connecting to 139.199.72.245 port 36000.
debug1: Connection established.
debug1: kex_exchange_identification: banner line 0: HTTP/1.1 502 Server UnReachable
debug1: kex_exchange_identification: banner line 1: proxy-agent: VM-129-13-centos

两个致命问题同时暴露：

🎯 坑 6（真正原因）：企业网络的 SSH 代理拦截

问题一：默认端口不是 22

开发容器的 /etc/ssh/ssh_config.d/ 中有配置把 SSH 默认端口改成了 36000。这是企业内部的 SSH 跳板端口。

问题二：HTTP 代理拦截了 SSH 流量

容器的所有外网 SSH 连接都经过一个 HTTP 代理（proxy-agent）。这个代理只能理解 HTTP 协议，收到 SSH 握手包后直接返回了 502 Server UnReachable。

这就是 kex_exchange_identification 错误的真正原因——SSH 客户端期望收到 SSH 版本字符串（如 SSH-2.0-OpenSSH_9.6），结果收到的是 HTTP/1.1 502。

🔑 核心教训：企业内网环境（蓝盾构建机、AnyDev 开发容器等）的网络出口通常都有 HTTP 代理，SSH 流量会被拦截。这个限制在服务端完全看不到——因为 SSH 连接压根没到服务器，在代理那一层就被挡了。

 

✅ 最终方案：本地 Mac 一键部署

对于纯静态个人网站，最简单可靠的方案就是本地执行部署脚本：

#!/bin/bash
set -e

SERVER="ubuntu@139.199.72.245"
KEY="~/.ssh/light_server.pem"

# 1. rsync 同步
rsync -avz --delete \
  --exclude='.git' --exclude='deploy.sh' \
  -e "ssh -i $KEY" ./ $SERVER:/var/www/my-space/

# 2. 重载 Nginx
ssh -i $KEY $SERVER 'sudo nginx -t && sudo systemctl reload nginx'

# 3. 健康检查
curl -s -o /dev/null -w "%{http_code}" https://yeranyang.com

配合一个 shell alias，实现 push + 部署一条命令搞定：

alias deploy="git push && ./deploy.sh"
 

🛡️ 顺便完成的服务器安全加固

虽然蓝盾方案没跑通，但排查过程中顺便把服务器安全加固了一遍。这些对任何公网服务器都是必备的：

配置 命令 说明
fail2ban sudo apt install -y fail2ban 自动封禁暴力破解 IP
禁止密码登录 PasswordAuthentication no 只允许密钥认证
MaxStartups MaxStartups 30:50:80 提升并发认证连接上限

💡 关于 EdgeOne：它只代理 HTTP/HTTPS 流量（80/443 端口），SSH 的 22 端口仍然直接暴露在公网，EdgeOne 保护不了。SSH 安全只能靠服务端自身配置。

 

📝 总结：踩坑经验清单

1. ssh -v 是排查一切 SSH 问题的第一步

别猜，开 verbose 模式看详细日志。一个晚上的排查工作，一条 ssh -v 就能定位。它会告诉你连的是哪个端口、走没走代理、在哪一步断开的。

2. 企业内网出口有代理，SSH 可能被拦截

蓝盾构建机、AnyDev 开发容器等企业环境的外网流量通常经过 HTTP 代理。SSH 协议无法通过 HTTP 代理，会被直接拦截。服务端日志看到的只是 "Connection closed [preauth]"，完全看不出是代理拦截。

3. CI/CD 传递多行密文用 Base64 编码

SSH 私钥、证书等多行内容，在 CI/CD 变量中传递时几乎必然会丢失换行。Base64 编码转成单行是最稳妥的做法。

4. 公网服务器的 SSH 安全是 Day 1 的事

服务器上线后几乎立刻就会被扫描。关密码登录 + 装 fail2ban，这两件事应该写进服务器初始化脚本里，而不是等出了问题再补。

5. 简单场景用简单方案

纯静态站点 + 个人项目，一个 deploy.sh 就够了。CI/CD 是好东西，但如果你的场景不需要多人协作、不需要构建流程、不需要自动化测试，一个 shell 脚本可能比一套流水线更合适。

 

🗺️ 附：排查路径全景图

① 蓝盾凭证变量不可用 → 改用文本框
② SSH 私钥换行被吃 → Base64 编码
③ 密钥格式正确仍连不上 → 检查服务器日志发现暴力破解
④ 安全加固后仍连不上 → 排除所有服务端因素
⑤ 换到开发容器也一样 → 怀疑网络链路
⑥ ssh -v 一把定位：HTTP 代理拦截了 SSH 流量
⑦ 放弃 CI/CD → 本地脚本部署 → 完美运行 ✅

回过头看，这个问题的根因其实就一句话：企业内网的 HTTP 代理拦截了 SSH 协议。但因为错误信息（kex_exchange_identification: Connection closed by remote host）太过笼统，很容易被引导到错误的排查方向。

下次遇到 SSH 连接问题，第一件事就是 ssh -v。记住了。

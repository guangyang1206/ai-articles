01 / 06

🚢 CI/CD 配置陷阱

蓝盾流水线显示「部署成功」，但登录服务器无响应。


现象很奇怪——
流水线日志明明报「连接成功，文件已传输」，
手动 SSH 却持续超时，连不上服务。


排查后发现根因：

BKS_SSH_PORT 变量指向 SSH 默认 22 端口，
而实际服务器 SSH 监听在 2222 端口。
CI/CD 的 SSH 配置被构建变量的默认值覆盖了。

解决方案：

在蓝盾流水线中明确设置 SSH_PORT=2222，
并在部署脚本中显式引用 $BKS_SSH_PORT 变量。

1/6 @Guang Yang
02 / 06

🔥 防火墙规则遗漏

手动 SSH 仍然频繁超时，TCP 端口不可达。


telnet ip 2222 无响应，
nmap -sS ip -p 2222 显示 filtered。


检查 firewall-cmd 后发现：

# 当前规则：缺少 2222 端口
firewall-cmd --list-all
# → public (active)
# ports: 80/tcp 443/tcp

解决方案：

firewall-cmd --add-port=2222/tcp --permanent
firewall-cmd --reload
⚠️ 添加新端口后一定要 --reload，否则规则不会生效。
2/6 @Guang Yang
03 / 06

🔒 SELinux 403 之谜

Nginx 配置完毕，访问却返回 403 Forbidden。


权限没错，owner 没错，目录可读——
浏览器就是固执地返回 403。


检查 SELinux 日志：

# /var/log/audit/audit.log 中看到：
type=AVC msg=audit(...): avc: denied { getattr }
for pid=... comm="nginx"
scontext=system_u:system_r:httpd_t:s0
tcontext=unconfined_u:object_r:default_t:s0

文件从其他路径复制过来，
SELinux 上下文标记为 default_t，
Nginx 的 httpd_t 域无权读取。

修复：恢复目录 SELinux 上下文
restorecon -Rv /var/www/static-site
3/6 @Guang Yang
04 / 06

💥 502 Bad Gateway

403 刚解决，又迎来 502 Bad Gateway。


Nginx 已经能提供静态文件，
但代理转发到后端服务时报 502。


排查链路：


① 检查后端服务是否运行 → ✓ 正常

② 检查 Nginx proxy_pass 配置 → 找到了

# 错误的配置：
location /api/ {
  proxy_pass http://localhost:3000; ← 少了 /
}

③ Nginx proxy_pass 末尾有无 / 行为完全不同：

带 / → 去掉匹配前缀后转发
不带 / → 将完整 URI 原样转发
加了 / 后，502 秒变 200。
4/6 @Guang Yang
05 / 06

🛡️ 服务器被爆破攻击

系统刚跑起来，auth.log 里全是爆破记录。


上线不到 1 小时，
journalctl -u sshd 显示来自 10+ 个 IP 的暴力破解尝试：

Failed password for root from
192.168.x.x port 51722 ssh2
→ 每分钟上百次尝试

加固方案：


① 关闭密码登录，启用 密钥认证

# /etc/ssh/sshd_config
PasswordAuthentication no
PubkeyAuthentication yes

② 安装 fail2ban 自动封禁

[sshd]
enabled = true
maxretry = 3
bantime = 3600
配置完成后立即生效，爆破 IP 被封禁后，
auth.log 安静如初。
5/6 @Guang Yang
